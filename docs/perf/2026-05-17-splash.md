# Wave 19 — full-page splash + eager background prefetch

Date: 2026-05-17
Branch: `perf/imgproxy-on-demand`
Companion to: `docs/perf/2026-05-17-feel-tuning.md`

## Goal

Three asks from the user (translated from Ukrainian):

1. There must be a FULL-PAGE loading screen hiding the initial photo
   loading on `/a/<token>` pages.
2. The cover photo must load FIRST.
3. Even if the user hasn't scrolled down yet, photo downloading
   should continue in the background.

Wave 18 already shipped cover-first preload (`<link rel="preload"
as="image" fetchPriority="high">` in the page head) and srcset, but
left two regressions visible:

- Below-the-fold tiles were `loading="lazy"`, so they only started
  fetching once the user scrolled. On a slow imgproxy, the first
  tap on row 9 took ~1 second to paint because the request started
  cold.
- There was no full-page overlay — the user saw a stagger of
  thumbhash placeholders being replaced by photos, which reads as
  busy / amateur rather than cinematic.

### Commit 1 — eager background prefetch

`PhotoTile.priority` is now `"high" | "low" | "lazy"` (was a
boolean). The renderer passes `"high"` for the first 32 tiles and
`"low"` for the rest. The `"low"` value maps to
`fetchPriority="low" + loading="eager"` (previously
`loading="lazy"`).

The browser starts every tile download immediately. Native
6-per-origin throttling on HTTP/1.1 dev means the actual on-wire
order respects priority — above-the-fold tiles drain first, then
the queue refills with `priority="low"` requests. By the time the
viewer scrolls past row 4, those tiles are already painted from
the local cache.

Only `"high"` tiles register with the page-load progress counter;
counting `"low"` tiles would gate the splash dismissal on tiles
the viewer can't see yet (defeats the purpose of the splash).

### Commit 2 — full-page splash overlay

`PageSplash.tsx` (new client component) renders a fixed-position
black overlay at z-100. Visual: a rose-tinted dual-ring pulse
(rings cross-fade on a 1.8s cycle, offset by 600ms) plus the
`gallery.divass.space` wordmark below in 10px uppercase. The
backdrop is `#0a0a0a` (matches the existing dark theme).

Dismiss logic, encoded in the pure `shouldDismissSplash` predicate:

```text
dismiss when:
  enabled is false  → empty album, render nothing
  OR msSinceMount >= 6000ms  → hard ceiling, never trap
  OR (msSinceMount >= 350ms                    [min visible]
      AND cover has resolved (if registered)   [LCP gate]
      AND first min(8, registered) tiles loaded)
```

The 8-tile target is decoupled from the progress-bar cap (32) — the
splash needs to drop quickly to feel responsive; the progress bar
keeps tracking the full above-fold set. Small albums (fewer than 8
photos) collapse the target to "all registered tiles" so the splash
doesn't wait forever for tiles that don't exist.

Hard ceiling = 6s: long enough for a cold-imgproxy cover on a slow
3G first paint, short enough that a dead network produces a
recoverable failure mode.

`prefers-reduced-motion: reduce` swaps the two pulsing rings for a
single static ring (still rose-tinted, still glowing). The splash
itself still appears and dismisses — it's the *pulse* that's
suppressed, not the presence.

### Pure-helper unit tests

`tests/components/PageLoadProgress.test.tsx` grew from 16 cases to
24. The 8 new cases cover:

- empty album → dismiss immediately
- min-visible-time floor (50ms = no dismiss even when ready)
- cover-loading gate (8/8 tiles ready, cover pending → wait)
- normal happy path (cover + 8 tiles → dismiss)
- small album → target collapses to `registered`
- hard timeout escape (0 loaded, 6000ms → dismiss anyway)
- cover-only page (no tiles → dismiss after cover load)
- no-cover layout-fallback path (ignores cover gate)

Plus 2 reducer tests for the new `registerCover` / `loadedCover`
idempotency: replays return the same reference so React subscribers
don't churn on a duplicate onLoad/onError pair.

Total: **+10 tests**, suite at 396 passing (was 386).

## Splash visual decision

Picked the dual-ring pulse + wordmark instead of:

- **Loading-bar at top**: already exists (`PageLoadProgress`), would
  duplicate work and look noisy with two bars.
- **Album title fading in**: requires plumbing title text into the
  splash and would need a different splash per album, which fights
  the "calm, always-the-same first impression" feel.
- **No visual cue, just black**: looked dead, especially on a stalled
  imgproxy where the splash holds for ~2-3s.

The dual-ring pulse is small (40×40 px), centered, and reads as
"alive but not anxious". The wordmark grounds it. Dark backdrop +
rose accent keeps it inside the existing cinematic vocabulary
(`cover-kenburns`, `bg-rose-400/90` accent line, `bg-bg #0a0a0a`).

## Bench expectations

The script `scripts/page-bench.ts` is unchanged. The numbers it
captures shift as follows when re-run on a real album:

- **first photo paint**: improves slightly (cover preload was already
  in place; the new splash doesn't block paint, it overlays it).
- **time to 50% images**: should drop noticeably on cold imgproxy —
  below-the-fold tiles now start fetching at t≈0 instead of waiting
  for scroll.
- **time to 100% images**: should drop on warm imgproxy because the
  browser keeps the connection queue saturated from the start. On
  cold imgproxy, the imgproxy concurrency=10 semaphore is now the
  binding constraint (one fewer waste cycle of "browser idle while
  not scrolled").
- **image bytes / total bytes**: should grow for a viewer who never
  scrolled (we now download tiles they wouldn't have seen). For a
  viewer who scrolls, identical to baseline.

The benchmark harness was NOT re-run for this wave — it requires the
local dev stack (Postgres + MinIO + worker + Next), and prod-mode
numbers from `dev-prod.bat` are more representative of what a real
viewer feels. The user should re-run `npx tsx scripts/page-bench.ts
--label wave-19 --runs 3` against a real album and compare to the
"After W1-W5" column in `2026-05-17-feel-tuning.md`.

## What was NOT done

- The "render the splash from server-side too" option in the brief —
  client-only splash was the cleaner of the two. The minimum
  350ms floor means there is no flash even on bfcache restores;
  rendering a server-side `#gh-splash` div would add SSR/hydration
  complexity for no perceptible improvement. PageSplash does still
  defensively `document.getElementById("gh-splash")?.remove()` so
  if we ever add server-side companion markup, it cleans up cleanly.
- Splash variant per page (favorites, locked, password) — only the
  main share page mounts PageSplash, since favorites uses the same
  shell. Password / locked pages don't render the shell so the
  splash never appears there.
