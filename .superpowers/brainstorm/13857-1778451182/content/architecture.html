Share link flow Client side

1
Client opens /a/Hk7eRq8x
Middleware reads token. If expired / invalid — 404. If password-protected — show password gate.
2
Issue anonymous viewer_id
If no gh_viewer cookie → generate UUID, set HttpOnly cookie scoped to share path. This is how we persist "which photos you liked" across sessions on the same device.
3
SSR renders gallery
Server fetches photos + this viewer's favorites in one query. Page returns with hearts already filled.
4
Double-click = like
Server action: toggleFavorite(token, photoId). Upserts favorites row keyed on (token, photo, viewer). Optimistic UI update.
5
Export
API route streams ZIP. Reads photo records, fetches from MinIO, pipes through archiver with chosen variant (web / original) and filters (all / favorites only).

Admin auth + upload Server side

1
Login at /admin/login
Single admin (you). Email + password (argon2 hash). On success → iron-session cookie, 30-day refresh.
2
Middleware guards /admin/*
No session → redirect to login. Edge runtime check.
3
Upload photos
Client requests presigned PUT URLs (one per file) → uploads direct to MinIO. After all uploaded, calls finalizeUpload(albumId, manifest) which inserts photo rows + queues derivative generation.
4
Derivative pipeline
Background job (pg-boss on same Postgres) reads original, generates 3 variants with sharp: thumb 400px, web 1600px, large 2400px. Writes back to MinIO. Marks photo "ready".
5
Generate share link
Click "Share" → new share_links row with random 12-char token, optional password / expiry. Returns gallery.divass.space/a/{token}.
